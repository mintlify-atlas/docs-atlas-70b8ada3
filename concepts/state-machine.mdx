Every major entity in B2B Import ERP вҖ” clients, requirements, quotes, jobs, invoices, and more вҖ” has a formally defined lifecycle expressed as a state machine. Instead of allowing free-form status updates, the platform only permits transitions that are explicitly declared in the Master State & Transition Matrix. This design eliminates invalid data states, makes the system auditable, and provides a single authoritative source of truth for business process enforcement.

<Note>
  All state names are in **Spanish** because B2B Import ERP is built for Spanish-speaking markets in Latin America. Database column values use uppercase snake_case codes (e.g., `"BORRADOR"`, `"EN_REVISION"`, `"APROBADA"`, `"RECHAZADA"`, `"VENCIDA"` for quotes), while this document uses human-readable Spanish labels for clarity.
</Note>

---

## Global Rule

> **No state change may ever be applied directly in the database.** All transitions must be executed through domain services.

Every state transition вҖ” regardless of entity type вҖ” must:

1. **Validate permissions** вҖ” the acting user must hold the required RBAC permission for the transition (e.g., `quotes.approve` to move a quote from `Enviada` вҶ’ `Aprobada`).
2. **Record an audit log entry** вҖ” the `audit_log` table captures `old_values`, `new_values`, `user_id`, `tenant_id`, `ip_address`, and `user_agent`.
3. **Emit a domain event** вҖ” downstream listeners (notifications, recalculations, integrations) react to the event asynchronously.
4. **Generate alerts if applicable** вҖ” certain transitions trigger `ALERTA` records (e.g., a quote reaching `Vencida`, a job moving to `Suspendido`).
5. **Update related dates** вҖ” timestamps such as `started_at`, `completed_at`, and `closed_at` are set automatically by the transition service, not by the caller.

## Client States

Clients progress from initial prospecting through active engagement to eventual archival.

```
Prospecto в”Җв”ҖвҶ’ Activo в”Җв”ҖвҶ’ Inactivo в”Җв”ҖвҶ’ Archivado
                в”Ӯ              вҶ‘
                в””в”Җв”ҖвҶ’ Bloqueado в”Җв”ҳ
                       в”Ӯ
                       в””в”Җв”ҖвҶ’ Activo  (unblock)
```

| Transition              | Allowed | Notes                                    |
| :---------------------- | :-----: | :--------------------------------------- |
| Prospecto вҶ’ Activo      | вң…      | First commercial engagement confirmed    |
| Activo вҶ’ Inactivo       | вң…      | No recent activity                       |
| Activo вҶ’ Bloqueado      | вң…      | Credit or compliance hold                |
| Bloqueado вҶ’ Activo      | вң…      | Hold resolved                            |
| Inactivo вҶ’ Activo       | вң…      | Re-engagement                            |
| Inactivo вҶ’ Archivado    | вң…      | Long-term dormancy                       |
| Prospecto вҶ’ Archivado   | вқҢ      | Not permitted вҖ” must activate first      |
| Bloqueado вҶ’ Archivado   | вқҢ      | Not permitted вҖ” must unblock first       |

## Requirement States

Requirements follow the full commercial-to-operational pipeline. Cancellation is an exit from any state.

```
Registrado
    вҶ“
DiagnГіstico          вҶҗ requires: assigned responsible user
    вҶ“
Visita TГ©cnica       вҶҗ requires: diagnosis record present
    вҶ“
CotizaciГіn           вҶҗ requires: sufficient technical information
    вҶ“
Aprobado             вҶҗ requires: approved quote linked
    вҶ“
En EjecuciГіn         вҶҗ requires: job (Trabajo) created
    вҶ“
Cerrado

Any state в”Җв”ҖвҶ’ Cancelado
```

| Transition                        | Validation Condition                        |
| :-------------------------------- | :------------------------------------------ |
| Registrado вҶ’ DiagnГіstico          | Responsible user assigned                   |
| DiagnГіstico вҶ’ Visita TГ©cnica      | Diagnosis record exists                     |
| Visita TГ©cnica вҶ’ CotizaciГіn       | Sufficient technical information captured   |
| CotizaciГіn вҶ’ Aprobado             | An approved quote (`Aprobada`) is linked     |
| Aprobado вҶ’ En EjecuciГіn           | A job (`Trabajo`) has been created          |
| Any вҶ’ Cancelado                   | No preconditions вҖ” available at any stage   |

## Quote States

Quotes move from drafting through review and dispatch to a final resolution state. Once resolved, they are immutable.

```
Borrador в”Җв”ҖвҶ’ En RevisiГіn в”Җв”ҖвҶ’ Enviada в”Җв”ҖвҶ’ Aprobada  (final)
   в”Ӯ                              в”Ӯ
   в”Ӯ                              в”ңв”Җв”ҖвҶ’ Rechazada  (final)
   в”Ӯ                              в””в”Җв”ҖвҶ’ Vencida    (final)
   в”Ӯ
   в””в”Җв”ҖвҶ’ Cancelada  (final)
```

| State       | Terminal | Notes                                              |
| :---------- | :------: | :------------------------------------------------- |
| Borrador    | No       | Internal draft, not yet submitted for review       |
| En RevisiГіn | No       | Under internal approval review                     |
| Enviada     | No       | Sent to client, awaiting response                  |
| Aprobada    | **Yes**  | Client accepted вҖ” triggers requirement advancement |
| Rechazada   | **Yes**  | Client declined                                    |
| Vencida     | **Yes**  | Passed expiry date without a response              |
| Cancelada   | **Yes**  | Withdrawn before dispatch                          |

## Job (Trabajo) States

Jobs represent the operational execution of an approved requirement. Suspension and re-activation are permitted mid-execution.

```
Pendiente в”Җв”ҖвҶ’ Programado в”Җв”ҖвҶ’ En EjecuciГіn в”Җв”ҖвҶ’ Finalizado в”Җв”ҖвҶ’ Entregado в”Җв”ҖвҶ’ Cerrado
                                   в”Ӯ
                                   в”ңв”Җв”ҖвҶ’ Suspendido в”Җв”ҖвҶ’ En EjecuciГіn  (resume)
                                   в”Ӯ
Pendiente / Programado в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв””в”Җв”ҖвҶ’ Cancelado
```

| Transition                    | Validation Condition                            |
| :---------------------------- | :---------------------------------------------- |
| Pendiente вҶ’ Programado        | Responsible user assigned; scheduled date set   |
| Programado вҶ’ En EjecuciГіn     | Actual start date (`inicio_real`) recorded      |
| En EjecuciГіn вҶ’ Suspendido     | No precondition вҖ” available at any time         |
| Suspendido вҶ’ En EjecuciГіn     | No precondition вҖ” resume any time               |
| En EjecuciГіn вҶ’ Finalizado     | All activities are in `Completada` state        |
| Finalizado вҶ’ Entregado        | Delivery record (`Acta Entrega`) uploaded       |
| Entregado вҶ’ Cerrado           | No pending activities remain                    |
| Pendiente вҶ’ Cancelado         | Available before scheduling                     |
| Programado вҶ’ Cancelado        | Available before execution starts               |

## Activity States

Individual activities within a job follow a parallel lifecycle to the job itself.

```
Pendiente в”Җв”ҖвҶ’ Programada в”Җв”ҖвҶ’ En EjecuciГіn в”Җв”ҖвҶ’ Completada
                                   в”Ӯ
                                   в””в”Җв”ҖвҶ’ Suspendida в”Җв”ҖвҶ’ En EjecuciГіn  (resume)

Pendiente в”Җв”ҖвҶ’ Cancelada
```

## Invoice States

Invoices move from pending through payment to either a paid or annulled terminal state. Partial payments are tracked as an intermediate state.

```
Pendiente в”Җв”ҖвҶ’ ParcialmentePagada в”Җв”ҖвҶ’ Pagada  (final)
    в”Ӯ               в”Ӯ
    в”Ӯ               в””в”Җв”ҖвҶ’ Vencida
    в”Ӯ
    в””в”Җв”ҖвҶ’ Vencida
    в”Ӯ
    в””в”Җв”ҖвҶ’ Anulada  (final)
```

| State                | Terminal | Notes                                      |
| :------------------- | :------: | :----------------------------------------- |
| Pendiente            | No       | Issued but no payment received             |
| ParcialmentePagada   | No       | One or more partial payments recorded      |
| Pagada               | **Yes**  | Fully settled вҖ” no further changes allowed |
| Vencida              | No       | Past due date without full payment         |
| Anulada              | **Yes**  | Voided before full payment                 |

## What Happens on Every Transition

The domain service that executes a transition always performs the following sequence atomically:

```ts
// Pseudo-code вҖ” domain transition service pattern
async function transitionState(entityId, targetState, actingUserId) {
  // 1. Load current entity and validate the transition is allowed
  const entity = await repo.findById(entityId);
  assertTransitionAllowed(entity.state, targetState);

  // 2. Check RBAC permission for this specific transition
  await assertPermission(actingUserId, requiredPermission(targetState));

  // 3. Validate domain preconditions (responsible user, linked records, etc.)
  await validatePreconditions(entity, targetState);

  // 4. Apply the state change and update related timestamps
  await repo.updateState(entityId, targetState);

  // 5. Write audit log entry
  await auditLog.record({ entityId, from: entity.state, to: targetState, actingUserId });

  // 6. Emit domain event for downstream consumers
  await eventBus.emit(`${entity.type}.state_changed`, { entityId, targetState });

  // 7. Create alert records if the transition triggers one
  await alertService.maybeCreate(entity, targetState);
}
```

<Warning>
  If a transition is not defined in the Master State & Transition Matrix, it must **not** be implemented, inferred, or invented. Stop and request a formal definition before writing any code.
</Warning>
